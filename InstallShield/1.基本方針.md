# C#アプリ（①通常版 / ②Secure版）分離対応 設計まとめ

---

# 1. 背景

既存システムとして以下が存在：

* ①クライアント（通常版）
* ①サーバー（通常版）

今回、新たに以下を追加する：

* ②クライアント（新機能対応版 / VisAP）
* ②サーバー（新機能対応版 / VisAP）

---

# 2. 要件

## 2.1 共存要件

* ①と②を同一PC上で共存可能にする

## 2.2 通信制御要件

* ①クライアント → ①サーバーのみ接続可能
* ②クライアント → ②サーバーのみ接続可能
* ①と②の相互接続は禁止

## 2.3 制約

* ①クライアント / ①サーバーは変更不可
* ②側のみ改修可能

## 2.4 将来要件

* 将来的に①と②を統合予定
* 製品名・ブランドは共通化したい

---

# 3. 設計方針

## 3.1 基本方針

* 表示上は同一製品（MyApp）
* 内部的には「Edition」で識別

## 3.2 Edition設計

```text
①通常版   → edition = STD
②Secure版 → edition = VISAP
```

---

# 4. 通信設計

## 4.1 採用方式

* SOAP Headerに `edition` を付与
* サーバー側で判定

---

## 4.2 ②クライアント

SOAP通信時に以下を付与：

```text
X-MyApp-Edition: VISAP
```

またはSOAP Header内：

```xml
<Edition>VISAP</Edition>
```

---

## 4.3 ②サーバー

### 受信時処理

* `edition=VISAP` がある → 許可
* ない → 拒否

### 判定ロジック例

```csharp
if (request.Edition != "VISAP")
{
    throw new Exception("このサーバーはVisAP版専用です");
}
```

---

## 4.4 通信制御の結果

| 通信パターン | 結果              |
| ------ | --------------- |
| ② → ②  | OK              |
| ① → ②  | NG（②サーバーで拒否）    |
| ② → ①  | NG（後述の新メソッドで防止） |
| ① → ①  | OK（既存のまま）       |

---

# 5. 新メソッド追加（②サーバー）

## 5.1 目的

* ②クライアントが接続先を事前確認するため
* ② → ①誤接続防止

---

## 5.2 メソッド例

```csharp
GetCompatibilityInfo()
```

戻り値例：

```json
{
  "product": "MyApp",
  "edition": "VISAP",
  "protocolVersion": 1
}
```

---

## 5.3 ②クライアント処理

1. 接続前に `GetCompatibilityInfo()` を呼ぶ
2. `edition == VISAP` を確認
3. 不一致なら接続中止

---

# 7. InstallShield対応方針

## 7.1 共存のために分離する項目

* ProductCode（必須）
* UpgradeCode（推奨：分離）
* インストールフォルダ
* EXE名
* サービス名（サーバー）
* レジストリキー
* 設定ファイル保存先
* ショートカット名

---

## 7.2 同一でもよい項目

* 製品ブランド名（MyApp）
* 会社名
* UI表示名（一部）

※ただしアンインストール一覧で識別できるように補助名推奨

---

## 7.3 フォルダ例

```text
C:\Program Files\Company\MyApp\Standard
C:\Program Files\Company\MyApp\VisAP
```

---

# 8. 実装方針まとめ

## 必須対応

* SOAP Headerで `edition=VISAP` を送信
* ②サーバーで受信チェック
* ②サーバーに新メソッド追加（接続確認用）

## 推奨対応

* 接続先URL分離
* エラーメッセージ明確化
* 定数管理（VISAP文字列）

---

# 9. リスクと対策

| リスク              | 対策                |
| ---------------- | ----------------- |
| ①クライアントが②サーバーに接続 | サーバー側でeditionチェック |
| ②クライアントが①サーバーに接続 | 新メソッドで事前判定        |
| 設定混在             | フォルダ/レジストリ分離      |
| 将来拡張困難           | edition設計採用       |

---

---

# 11. 最終方針

* 製品名は統一（MyApp）
* 内部識別は `edition`
* 通信はSOAP Headerで判定
* サーバー側で拒否制御
* 新メソッドで誤接続防止
* InstallShieldは内部識別を分離

---

# 12. 補足

* `edition=VISAP` は「新機能対応版」を意味する識別子
* 将来的に `SEC` などへ抽象化可能
* 現段階では VISAP で問題なし
